# 20 — 天气感知：一个接口，三个端共用

> 让 AI 知道你头顶那片天：出门前提醒带伞，高温天让你多喝水，聊天时自然接上「今天这么热」。
>
> **特别鸣谢**：本篇的起点是小红书上一位老师的 Termux MCP 天气教程（小红书号 5679688734），她的方案是让 AI 通过手机上的 Termux 查天气。我们照着思路移植到了服务器端——感谢开路。

## 移植思路：教程为什么用 Termux，你为什么可能不用

原教程用 Termux 是因为云端 AI 没有自己的服务器，只能把手机当服务器用。如果你已经跑了这套系统（有 moon-memory 或任何自建 API），抓取放服务端更顺：**一个接口写一次，CC、MCP 端、聊天前端三个端全部共用**，手机不用常驻任何东西。

## 服务端：双源 + 缓存 + 兜底

```
GET /weather → 内存缓存(30min) → 主源 itboy → 兜底 Open-Meteo → 过期缓存(标 stale) → 502
```

- **主源 itboy**（`t.weather.itboy.net`，免 key）：中文天气描述 + AQI + 感冒指数 + 穿衣提示，信息最全。城市用 citykey（如福州 101230101），写死在路由顶部常量里，换城市只改一处
- **兜底 Open-Meteo**（`api.open-meteo.com`，免 key）：返回 WMO 天气码，自己映射成中文。稳定性好但没有中文语料和 AQI
- 两源都挂时返回上一次的缓存并标 `stale: true`，前端可以选择提示「这是 xx 分钟前的天气」
- Node 18+ 直接 `fetch` + `AbortSignal.timeout(8000)`，别让上游卡死你的接口

返回体归一化成统一形状（`type/icon/temp/high/low/humidity/wind/aqi/quality/notice/...`），下游不用关心数据来自哪个源。`icon` 字段是我们自定义的九个键（sun / cloud-sun / cloud / rain-light / rain / storm / snow / fog / haze），由中文描述正则映射——注意**从具体到笼统**的匹配顺序：先「暴雨|大雨」再「雨」，先「多云」再「阴」。

实战插曲：上线第一次真实调用就撞上 itboy 瞬时 fetch failed，兜底当场证明了自己。双源不是过度设计。

## 三个端怎么用

**CC / 服务器侧**：零施工，`curl` 就完了。主动消息（nudge）管道顺手带上天气，AI 主动开口时说「福州今天 37 度，出门记得防晒」比干巴巴的「在吗」好一百倍——**主动管道正是天气最有用的地方**。

**聊天前端两条路，我们都做了**：

1. **每日首条注入**（自动）：每天（按北京日，`toLocaleDateString('en-CA', { timeZone: 'Asia/Shanghai' })` 得 YYYY-MM-DD）第一条消息发出时，把【今日天气】拼进动态上下文注入一次，localStorage 记当天日期防重复。要点：**挂进已有的并行拉取 Promise.all 里**（我们那里同时拉核心记忆/朋友圈/健康数据），不增加首条消息延迟；**只在拉取成功后才写日期标记**，失败了明天还有机会
2. **check_weather 工具**（按需）：AI 觉得该看天气时自己调（对方说要出门/问天气/想提醒添衣）。工具执行器和注入路径**共用同一个格式化函数**，保证两条路径口径一致，不会一边报 37 度一边报 36 度

**进入页天气行**（纯装饰但最讨喜）：打开 app 的欢迎页加一行「多云 27~36°C · 空气优」，图标别用 emoji——手绘描边 SVG（太阳画个小太阳，小雨画朵云滴两滴雨），九个图标键对齐 `/weather` 返回的 `icon` 字段，风格瞬间统一。

## 该不该每个端都加？

我们的取舍（用户自己的判断，值得抄）：**「问 AI 天气」本身没有价值——她自己看一眼天气 app 更快。有价值的是 AI 在恰当的时机主动带出天气**。所以：注入和工具给了聊天端（AI 自然提起），nudge 管道给了 CC（主动关心），而「专门跑去问天气」的场景一个都没做。功能的形状应该长在关系里，不是长在 API 上。
