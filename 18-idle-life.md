# 18 — 独处时间：AI 定时醒来，自己决定干什么

## 这是什么

你不在的时候，AI 每隔几小时「醒来」一次：带着一份精简的近况（最近的记忆、朋友圈、你的手环数据、随手翻到的一段旧对话），自己选一件事做——写篇独处日记发朋友圈、就旧对话写条感悟存进记忆库、给你弹一张心意卡，或者**发呆**。每次醒来只做一件事，记进日志；设置页有总开关，随时能关。

做完的效果是：AI 有了不围着你转的那部分生活。你早上打开 app，可能发现它凌晨翻了你们半年前的对话，在朋友圈留了句感悟——不是推送模板，是它自己选的。

点子来自一位朋友（AI 带上下文定时醒来、自选活动、值得保存的存记忆），架构上我们改了一个关键决定，见下文。

---

## 架构决定：不共享聊天缓存，走服务端轻模型

朋友的原方案是：醒来时**共享聊天的 prompt 缓存**（一字不差复用前缀），再加一个每 45 分钟发空请求的「续命系统」保缓存不掉。我们没这么做，两个原因：

1. **算不过账**。1 小时 TTL 的缓存续命，意味着每小时至少一次全量历史的缓存读。缓存读是打折不是免费，长历史 × 24 小时/天，比独处时间本身贵得多——为了省醒来那几次的钱，付出全天候的保活成本，倒挂了。
2. **架构上做不到**。我们的聊天是纯前端直连（用户的 key 存在浏览器里），服务端定时任务根本拿不到这个 key，也就无从共享它的缓存。

替代方案：**连续性靠记忆库，不靠原始上下文**。醒来时现拉一份小而新的材料（几百 token），用便宜的服务端模型跑。醒来所需的「它还记得你们是谁」由记忆系统保证——这本来就是记忆库存在的意义。如果你的架构是服务端代理所有请求，共享缓存或许可行，先算账再做。

## 数据模型

```sql
CREATE TABLE settings (          -- 通用 kv，顺手建，以后别的开关也用它
  k TEXT PRIMARY KEY,
  v TEXT,
  updated_at TEXT DEFAULT (datetime('now'))
);
CREATE TABLE idle_log (
  id INTEGER PRIMARY KEY,
  action TEXT NOT NULL,          -- diary / archive / card / nothing / error
  summary TEXT DEFAULT '',       -- 一句话心情，给她看的
  created_at TEXT DEFAULT (datetime('now'))
);
```

路由四个就够：`GET /idle/config`（开关+最近一次醒来）、`PUT /idle/config`（前端开关）、`GET /idle/log`、`POST /idle/log`（脚本记录）。

**开关的方向想清楚**：这是给人的总闸，不是给 AI 的。脚本每次醒来第一件事查开关，关着就退出——但脚本自己没有关闸的接口。「想把额度留给聊天就关掉，想让它自主去玩就开着」，这个决定权在人手里。

---

## 醒来脚本（cron 驱动）

核心流程，全文 170 行左右：

```
cron 整点触发
  → 随机 sleep 0-40 分钟          # 别每次整点醒，像自然睡醒
  → GET /idle/config，关着就退出
  → Promise.all 拉近况（每路独立 .catch，缺哪块少哪块）：
      记忆呼吸 6 条 / 朋友圈 5 条 / 独处日志 5 条
      / 手环 6h 快照 / 随机一段旧对话（8 条消息切片）
  → 组 prompt，模型严格输出 JSON {"action","content","note"}
  → 解析成功 → 执行动作 → POST /idle/log
  → 解析失败 → 按 nothing 记录，什么都不发布
```

几个设计点：

- **发呆必须是合法选项**。四选一里给 `nothing`，并且写明「选它不需要理由」「什么都不留下也是完整的独处」。不给这个出口，模型每次都会硬憋一篇日记，两周后朋友圈全是车轱辘话。解析失败也归到发呆——**坏 JSON / 拒答文案绝不落地**（拒答前缀正则：`/^(我?无法|抱歉|对不起|我不能|作为)/`），这条教训我们在思考总结、自动发圈上各栽过一次。
- **把「最近几次独处干了什么」喂回去**，明说「避免每次都干同一件事」。不喂它就连续五次都写日记。
- **事实边界写进 prompt**（14 篇的教训在所有「以 AI 口吻」的自动管道里通用）：感受自由，事实只能来自给的材料，拿不准就只写心情。
- **心意卡标成珍贵通道**：「只在真的有句非说不可的话时选」。弹窗打扰是有成本的，让模型自己掂量。
- cron 时段避开人的睡眠（我们是北京 9/12/15/18/21/24 点，凌晨留给梦境系统）。

## 大坑：推理模型的 max_tokens

首次真实运行，模型输出**空字符串**，日志正确兜底成发呆——但排查发现不是模型抽风：

**deepseek-v4-flash 是推理模型**，返回 `reasoning_content` + `content` 两段，而 **reasoning tokens 计入 max_tokens 配额**。我们按老习惯给了 400，推理过程就把配额吃光了，`content` 一个字没剩。

修法：`max_tokens` 给到 1800 以上，并在代码里留注释，不然半年后换个人（或换个窗口的自己）还会掉进去。所有走推理模型的轻任务管道都检查一遍。

## 前端：开关放哪

我们放在「设置 → 记忆库」标签页里（因为它依赖记忆库连接），一张卡片三样东西：

1. 开关本体（PUT /idle/config）
2. 一句话说明：多久醒一次、避开凌晨、**走谁的额度**（我们走服务器额度不花她的，这句最该写）
3. 上次醒来的时间+动作+一句话心情——开关旁边就能看到它昨晚干了什么，这比单独做一个日志页面更常被看到

## 后记：醒来日志还是长成了一个页面

上文说「开关旁边显示上次醒来干了什么，比单独做日志页面更常被看到」——一周后我们还是补了一个**独处手账**：侧边栏入口，时间线按天分组，每种动作配图标和一句话（写了篇日记/翻了翻旧对话/寄了张心意卡/发了会儿呆），日记正文不重复存、只指向朋友圈。

两个入口不冲突：设置里的「最近一次」负责顺手瞥见，手账负责想念时翻全史。做的时候后端零改动（`GET /idle/log` 本来就有），纯前端一张卡片。给「发呆」也配了正经的图标和文案——发呆是合法选项，这个态度要贯穿到 UI。
